System Design

Author: Naresh Kumar Sanda

CAP Theorem, Load Balancing, Sharding, and Message Queues.

1. CAP Theorem

In a distributed data store, you can only guarantee two out of the following three guarantees simultaneously:

  • Consistency: Every read receives the most recent write or an error. (All nodes see the same data at the same time).
  • Availability: Every request receives a non-error response, without the guarantee that it contains the most recent write.
  • Partition Tolerance: The system continues to operate despite an arbitrary number of messages being dropped/delayed between nodes.

Note: Because network partitions (P) are unavoidable in distributed systems, you usually have to choose between CP (Consistency over Availability) or AP (Availability over Consistency).

2. Load Balancing

Distributes incoming network traffic across a group of backend servers (Server Farm).

Algorithms

  • Round Robin: Distributes requests sequentially across the group of servers.
  • Least Connections: Sends the request to the server with the fewest active connections. Good for long-running connections like WebSockets.
  • IP Hash: Uses the client's IP address to determine which server receives the request. Useful for maintaining session persistence (Sticky Sessions).

L4 vs L7 Load Balancing

  • Layer 4 (Transport): Balances based on IP address and TCP/UDP port. Fast, but blind to the actual content (HTTP headers/URL).
  • Layer 7 (Application): Balances based on the HTTP request (URL path, cookies, headers). Slower, but allows smart routing (e.g., /api to backend, /images to CDN).

3. Database Scaling

Replication (Read Scaling)

Master-Slave architecture. All writes go to the Master, which replicates data to Slaves. Reads can be distributed among Slaves. Issue: Replication lag.

Sharding (Write Scaling)

Partitioning data across multiple databases (shards).

  • Key-Based (Hash) Sharding: Hash the ID (e.g., `user_id % num_servers`). Ensures even distribution, but adding/removing servers requires massive data re-shuffling.
  • Consistent Hashing: Solves the re-shuffling problem. Servers and keys are mapped to a conceptual "hash ring". Adding a node only redistributes keys from the immediate neighbor.
  • Directory-Based Sharding: Maintain a lookup table mapping a key to a specific database node. Flexible but adds a single point of failure.

4. Message Queues (Pub/Sub)

Used to decouple services, smooth out traffic spikes (buffering), and execute heavy tasks asynchronously.

RabbitMQ / SQS "Smart broker, dumb consumer". The broker tracks which messages were read (ACKs) and deletes them. Good for traditional task queuing.
Apache Kafka "Dumb broker, smart consumer". Messages are stored in an append-only log and persist for a configured time. Consumers keep track of their own "offset" (where they are in the log). Good for massive event streaming and replayability.

5. API Design Patterns

  • REST: Resource-oriented (Nouns). Standard HTTP verbs (GET, POST, PUT, DELETE). Stateless.
  • GraphQL: Client dictates exactly what data it wants. Prevents Over-fetching and Under-fetching. Uses a single POST /graphql endpoint.
  • gRPC: Uses Protocol Buffers (binary). Highly compressed and extremely fast. Excellent for internal microservice-to-microservice communication. Not natively supported by web browsers.
