Caching Strategies

Author: Naresh Kumar Sanda

Read/Write strategies, Eviction policies, and Redis use cases.

1. Cache-Aside (Lazy Loading)

The application is responsible for reading and writing to the cache and the database.

  • Read Flow: App checks cache. If hit, return data. If miss, app reads from DB, writes to cache, and returns data.
  • Write Flow: App writes directly to DB. App then invalidates (deletes) the cache entry.
  • Pros: Cache only contains data that is actually requested (saves memory). Node failures are handled gracefully (just results in DB reads).
  • Cons: Cache misses cause 3 network hops (Cache Check -> DB Read -> Cache Write), increasing latency.

2. Write-Through

The application writes to the cache, and the cache synchronously writes to the database.

  • Read Flow: App reads from cache. (Misses are handled similarly to Cache-Aside, often by the cache provider automatically fetching from DB).
  • Write Flow: App writes to Cache -> Cache writes to DB -> Return Success.
  • Pros: Data in the cache is never stale. High read performance.
  • Cons: Write operations are slower because they require two synchronous writes. Cache might get filled with data that is rarely read.

3. Write-Back (Write-Behind)

The application writes to the cache, and the cache asynchronously writes to the database at a later time (or in batches).

  • Pros: Extremely fast write performance. Reduces database load since writes are batched.
  • Cons: High risk of data loss. If the cache node crashes before the background write to the database occurs, the data is lost permanently. Complex to implement.

4. Eviction Policies

Caches are stored in RAM, which is expensive and limited. When the cache is full, an eviction policy determines which data to throw away.

LRU (Least Recently Used) Evicts the items that haven't been accessed for the longest time. The most common default policy.
LFU (Least Frequently Used) Evicts the items that are accessed the least often over a period of time. Requires keeping a counter for each item.
FIFO (First In, First Out) Evicts the oldest items in the cache, regardless of how often they are accessed.
TTL (Time To Live) Not strictly an eviction policy when full, but an expiration mechanism. Items automatically delete themselves after a set time (e.g., 60 seconds). Crucial for keeping Cache-Aside data fresh.

5. Redis vs Memcached

Both are popular, open-source, in-memory key-value stores. But they have different use cases.

Memcached

  • Extremely simple key-value store for strings/objects.
  • Multi-threaded architecture (can utilize multiple CPU cores out of the box).
  • No persistence (restarts = total data loss).

Redis (Remote Dictionary Server)

  • Advanced data structures (Strings, Lists, Sets, Hashes, Sorted Sets, Bitmaps).
  • Single-threaded (processes one command at a time, ensuring atomicity without locks).
  • Supports persistence (RDB snapshots, AOF logs) so data survives restarts.
  • Can be used for Pub/Sub messaging, Leaderboards (Sorted Sets), and Distributed Locks.
