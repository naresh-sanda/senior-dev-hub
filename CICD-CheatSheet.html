CI/CD Pipelines

Author: Naresh Kumar Sanda

Continuous Integration, Delivery, and modern deployment strategies.

1. Core Concepts

Continuous Integration (CI) The practice of merging all developer working copies to a shared mainline several times a day. Triggers automated build and unit tests to ensure the branch isn't broken.
Continuous Delivery (CD) An extension of CI. Ensures the codebase is always in a deployable state. Involves packaging the application (e.g., Docker image) and pushing it to an artifact repository, then optionally deploying to a staging environment. Requires manual approval to go to Production.
Continuous Deployment (CD) Goes one step further: every change that passes all stages of your production pipeline is automatically released to customers. No human intervention.

2. Deployment Strategies

Blue/Green Deployment

Run two identical production environments (Blue and Green). At any time, only one environment is live. When a new release is ready, you deploy to the idle environment, test it, and then switch the router (load balancer) to point to the new environment.

  • Pros: Zero downtime, instant rollback by switching the router back.
  • Cons: Requires double the infrastructure resources.

Canary Release

Deploy the new version to a small subset of servers or route a small percentage of user traffic (e.g., 5%) to the new version. Monitor for errors. If stable, gradually roll out to 100%.

  • Pros: Reduces risk. If there's a critical bug, only 5% of users are affected.
  • Cons: Complex to orchestrate routing and monitoring.

Rolling Update (Default in K8s)

Gradually replace old instances with new ones, one by one. K8s will wait for the new Pod's readiness probe to pass before killing an old Pod.

3. GitHub Actions Example

GitHub Actions uses YAML files located in .github/workflows/ to define CI/CD pipelines.

name: CI Pipeline

on:
  push:
    branches: [ "main" ]
  pull_request:
    branches: [ "main" ]

jobs:
  build-and-test:
    runs-on: ubuntu-latest
    steps:
      - name: Checkout code
        uses: actions/checkout@v3
        
      - name: Set up JDK 17
        uses: actions/setup-java@v3
        with:
          java-version: '17'
          distribution: 'temurin'
          
      - name: Build with Maven
        run: mvn clean package
        
      - name: Build and Push Docker Image
        if: github.event_name == 'push'
        run: |
          docker login -u ${{ secrets.DOCKER_USER }} -p ${{ secrets.DOCKER_PAT }}
          docker build -t myorg/myapp:${{ github.sha }} .
          docker push myorg/myapp:${{ github.sha }}

4. Jenkinsfile Example (Declarative)

pipeline {
    agent any
    tools {
        maven 'Maven 3.8'
        jdk 'JDK 17'
    }
    stages {
        stage('Checkout') {
            steps {
                checkout scm
            }
        }
        stage('Test') {
            steps {
                sh 'mvn test'
            }
        }
        stage('Build & Push Image') {
            when { branch 'main' }
            steps {
                withCredentials([usernamePassword(credentialsId: 'dockerhub', usernameVariable: 'USER', passwordVariable: 'PASS')]) {
                    sh 'docker build -t myapp:${GIT_COMMIT} .'
                    sh 'echo $PASS | docker login -u $USER --password-stdin'
                    sh 'docker push myapp:${GIT_COMMIT}'
                }
            }
        }
    }
}

5. Pipeline Best Practices

  • Build Once, Deploy Many: Never recompile code for different environments (dev/stage/prod). Build a single artifact (like a Docker image), and inject environment-specific configuration at runtime.
  • Fail Fast: Put the fastest tests (unit tests, static analysis, linting) at the very beginning of the pipeline. Don't waste time spinning up integration databases if the code doesn't even compile.
  • Immutable Infrastructure: Do not SSH into servers to tweak configurations. If a change is needed, modify the codebase (Terraform/Docker) and trigger the pipeline to replace the old server.
  • Automate Rollbacks: Ensure database migrations are backwards compatible so you can easily revert to the previous application version if something goes wrong.
