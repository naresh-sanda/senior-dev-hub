Design Patterns & Architecture

Author: Naresh Kumar Sanda

Gang of Four (GoF) patterns and Microservices Architecture, with Java examples and visual animations.

1. Creational Patterns

These patterns deal with object creation mechanisms, trying to create objects in a manner suitable to the situation.

Singleton

Ensures a class has only one instance and provides a global point of access to it.

public class DatabaseConnection {
    // volatile ensures multiple threads handle the singleton instance correctly
    private static volatile DatabaseConnection instance;

    private DatabaseConnection() {
        // Private constructor prevents instantiation
    }

    // Double-Checked Locking (Thread Safe)
    public static DatabaseConnection getInstance() {
        if (instance == null) {
            synchronized (DatabaseConnection.class) {
                if (instance == null) {
                    instance = new DatabaseConnection();
                }
            }
        }
        return instance;
    }
}

Factory Method

Provides an interface for creating objects in a superclass, but allows subclasses to alter the type of objects that will be created.

public interface Notification { void notifyUser(); }
public class SMSNotification implements Notification { public void notifyUser() { ... } }
public class EmailNotification implements Notification { public void notifyUser() { ... } }

public class NotificationFactory {
    public Notification createNotification(String channel) {
        if (channel.equals("SMS")) return new SMSNotification();
        if (channel.equals("EMAIL")) return new EmailNotification();
        throw new IllegalArgumentException("Unknown channel");
    }
}

Builder

Separates the construction of a complex object from its representation, allowing the same construction process to create various representations (Often implemented using Lombok's @Builder in modern Java).

User user = new User.UserBuilder()
    .firstName("Naresh")
    .lastName("Sanda")
    .age(30)
    .build();

2. Structural Patterns

These patterns explain how to assemble objects and classes into larger structures while keeping these structures flexible and efficient.

Adapter

Allows objects with incompatible interfaces to collaborate. (e.g., Making a Legacy JSON API work with a modern XML system).

public interface LightningPhone { void recharge(); }
public interface MicroUsbPhone { void useMicroUsb(); }

public class LightningToMicroUsbAdapter implements MicroUsbPhone {
    private final LightningPhone lightningPhone;

    public LightningToMicroUsbAdapter(LightningPhone lightningPhone) {
        this.lightningPhone = lightningPhone;
    }

    @Override
    public void useMicroUsb() {
        System.out.println("Adapter converting MicroUSB signal to Lightning...");
        lightningPhone.recharge();
    }
}

Facade

Provides a simplified interface to a library, a framework, or any other complex set of classes.

Example: Creating an OrderFacade that internally calls InventoryService, PaymentService, and ShippingService, hiding the complexity from the frontend controller.

Decorator

Attaches new behaviors to objects by placing these objects inside special wrapper objects that contain the behaviors. (Think: Java's InputStream wrappers).

3. Behavioral Patterns

These patterns are concerned with algorithms and the assignment of responsibilities between objects.

Strategy

Defines a family of algorithms, encapsulates each one, and makes them interchangeable. Strategy lets the algorithm vary independently from clients that use it.

public interface PaymentStrategy { void pay(int amount); }

public class CreditCardStrategy implements PaymentStrategy { ... }
public class PayPalStrategy implements PaymentStrategy { ... }

public class ShoppingCart {
    // The algorithm is injected dynamically!
    public void checkout(int amount, PaymentStrategy strategy) {
        strategy.pay(amount);
    }
}

Observer (Pub/Sub)

Defines a one-to-many dependency between objects so that when one object changes state, all its dependents are notified and updated automatically.

public class EventManager {
    private List<EventListener> listeners = new ArrayList<>();

    public void subscribe(EventListener listener) { listeners.add(listener); }
    
    public void notify(String eventType) {
        for (EventListener listener : listeners) {
            listener.update(eventType);
        }
    }
}

4. Microservices Architecture Patterns

Patterns for distributed systems, ensuring resiliency, scalability, and data consistency.

Circuit Breaker

Prevents a network or service failure from cascading to other services. If a downstream service fails repeatedly, the circuit "opens" and immediately returns an error (or fallback) without making the network call, giving the broken service time to recover.

Saga Pattern (Distributed Transactions)

Instead of a monolithic ACID transaction, a Saga is a sequence of local transactions. If a local transaction fails, the Saga executes a series of compensating transactions that undo the changes made by the preceding local transactions.

  • Choreography: Each local transaction publishes domain events that trigger local transactions in other services. (Decentralized, good for simple workflows).
  • Orchestration: An orchestrator (like Camunda or AWS Step Functions) tells the participants what local transactions to execute. (Centralized, good for complex workflows).

CQRS (Command Query Responsibility Segregation)

Separates the model used to update data (Commands) from the model used to read data (Queries). Often used with Event Sourcing.

  • Commands: Change state (POST, PUT, DELETE). Writes to a highly normalized relational database.
  • Queries: Read state (GET). Reads from a denormalized NoSQL database (like MongoDB or Elasticsearch) optimized for fast reading.
  • Data Synchronization: The Write DB publishes events to a message queue (Kafka), and the Read DB consumes them to update its state eventually.

API Gateway

A single entry point for all clients. It encapsulates the internal system architecture and provides an API that is tailored to each client. It handles routing, composition, and edge functions like authentication, rate limiting, and CORS.
