Spring Microservices

Author: Naresh Kumar Sanda

Spring Cloud, Gateway, Resiliency, and Distributed Patterns.

1. Service Discovery (Eureka / Consul)

Instead of hardcoding IP addresses, microservices register themselves with a central registry. Other services query the registry to find them dynamically.

  • @EnableEurekaServer: Turns a Spring Boot app into a registry.
  • @EnableDiscoveryClient: Tells a microservice to register itself.

Note: In Kubernetes, this is often handled natively by K8s Services and CoreDNS, making Eureka unnecessary in modern cloud-native stacks.

2. API Gateway Pattern

A single entry point for all clients. Handles routing, authentication, rate limiting, and CORS.

# Spring Cloud Gateway Configuration
spring:
  cloud:
    gateway:
      routes:
        - id: user_service_route
          uri: lb://USER-SERVICE # Uses Load Balancer to find the service via Eureka/Consul
          predicates:
            - Path=/api/users/**
          filters:
            - RewritePath=/api/users/(?<segment>.*), /$\{segment}

3. Circuit Breaker (Resilience4j)

Prevents cascading failures. If Service A calls Service B, and B is down, the Circuit Breaker "opens" and immediately fails future calls (or returns a fallback) without waiting for timeouts.

@Service
public class OrderService {

    @CircuitBreaker(name = "paymentService", fallbackMethod = "fallbackPayment")
    public PaymentResponse processPayment(OrderRequest request) {
        // HTTP call to Payment Service via RestTemplate or WebClient
        return restTemplate.postForObject("http://PAYMENT-SERVICE/pay", request, PaymentResponse.class);
    }

    // Fallback method must have the same signature + Throwable
    public PaymentResponse fallbackPayment(OrderRequest request, Throwable t) {
        return new PaymentResponse("FAILED", "Payment service is currently unavailable");
    }
}

Circuit States:

  • Closed: Everything is fine. Requests pass through.
  • Open: Failure threshold reached. Requests are blocked immediately.
  • Half-Open: After a timeout, allows a few test requests to check if the downstream service has recovered.

4. Distributed Transactions (Saga Pattern)

A standard database transaction (ACID) cannot span multiple microservices. Instead, use the Saga pattern.

  • A Saga is a sequence of local transactions.
  • If one step fails, the Saga executes Compensating Transactions to undo the previous steps.

Example: E-commerce Order

  1. Order Service creates Order (Status: PENDING).
  2. Inventory Service reserves stock. (If fails, abort Saga).
  3. Payment Service processes payment. (If fails, execute Compensation: Un-reserve stock in Inventory Service).
  4. Order Service updates Order (Status: APPROVED).

Implementation: Usually orchestrated using message brokers (Kafka/RabbitMQ) or orchestration engines like Camunda.

5. Distributed Tracing

When a request spans 5 different microservices, debugging an error is impossible without tracing.

  • Trace ID: A unique ID assigned to a request when it hits the API Gateway. Passed along to every downstream service via HTTP Headers.
  • Span ID: A unique ID for a specific operation/service within the overall Trace.

Tools: Spring Boot 3 uses Micrometer Tracing (previously Spring Cloud Sleuth). View traces in Zipkin or Jaeger.
