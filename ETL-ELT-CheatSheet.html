ETL & ELT Data Pipelines

Author: Naresh Kumar Sanda

A comprehensive guide to Data Engineering architectures, flow, and terminology.

1. ETL vs ELT

Data integration paradigms have shifted as cloud data warehouses became immensely powerful.

Concept ETL (Extract, Transform, Load) ELT (Extract, Load, Transform)
Flow Source -> Processing Engine (Spark/Python) -> Destination Source -> Destination Storage -> Transformed inside Destination via SQL
Compute Heavy Middle tier (e.g., Apache Spark, AWS Glue, DataStage) Destination tier (e.g., Snowflake, BigQuery, Redshift)
Best Used For Legacy systems, strict compliance rules before landing data, complex programmatic logic. Modern cloud stacks, agile data teams, leveraging SQL for transformations (dbt).
Data Loss Risk If transformation fails, raw data might not be persisted. Raw data is safely landed first. Easy to re-run transformations.

2. Data Storage Architectures

Data Warehouse

Highly structured, schema-on-write repository. Optimized for complex analytical queries (OLAP) using SQL. Examples: Snowflake, Amazon Redshift, Google BigQuery.

Data Lake

Schema-on-read repository storing raw data in its native format (structured, semi-structured, unstructured). Highly scalable and cheap. Examples: Amazon S3, Azure Data Lake (ADLS), Google Cloud Storage (GCS).

Data Lakehouse

Combines the flexibility and cheap storage of a Data Lake with the ACID transactions and governance of a Data Warehouse. Powered by open table formats. Examples: Databricks (Delta Lake), Apache Iceberg, Apache Hudi.

3. Data Processing Architectures

Lambda Architecture

Designed to handle massive quantities of data by taking advantage of both batch and stream-processing methods.

  • Batch Layer: Manages the master dataset (immutable, append-only) and pre-computes batch views. Highly accurate but high latency.
  • Speed Layer (Stream): Processes recent data in real-time to provide low-latency views. Less accurate (may miss late arrivals).
  • Serving Layer: Merges results from the Batch and Speed layers to answer user queries.
  • Drawback: You have to maintain two separate codebases (one for batch, one for streaming).

Kappa Architecture

An alternative to Lambda that removes the batch layer entirely.

  • Treats everything as a stream. Data is an immutable stream of events (e.g., in Apache Kafka).
  • If you need to recompute historical data, you simply replay the stream from the beginning using the same codebase.
  • Drawback: Requires robust and expensive streaming infrastructure.

4. Core Pipeline Concepts & Terminology

DAG (Directed Acyclic Graph) A conceptual representation of a pipeline's tasks and dependencies. "Directed" means data flows in one direction; "Acyclic" means no infinite loops. Used by orchestrators like Airflow and Dagster.
Idempotency A pipeline is idempotent if running it once has the exact same effect as running it multiple times. Crucial for handling failures and retries without duplicating data.
CDC (Change Data Capture) A technique to identify and capture changes made to data in a database (Inserts, Updates, Deletes) and deliver those changes in real-time to a downstream system (e.g., using Debezium).
SCD (Slowly Changing Dimensions) Techniques for managing how historical data is updated.
Type 1: Overwrite old data.
Type 2: Add a new row with effective/expiration dates (keeps history).
Backfilling Processing historical data that was missed or needs to be re-processed due to a bug fix or new logic deployment.
Data Lineage Tracking data as it flows from its origin to its destination, detailing transformations along the way. Essential for debugging and compliance.

5. Data Modeling Strategies

Star Schema (Kimball)

The standard for data warehouses. Denormalized for fast reads.

  • Fact Tables: Contain measurable, quantitative data (e.g., `sales_amount`, `discount`). Usually massive and append-only.
  • Dimension Tables: Contain descriptive attributes related to the facts (e.g., `customer_name`, `store_location`). Smaller and joined to facts.

Medallion Architecture (Databricks)

Standard data flow design in a Lakehouse.

  • 🥉 Bronze (Raw): Raw, unvalidated data directly from sources. Append-only.
  • 🥈 Silver (Cleaned): Filtered, cleaned, augmented, and normalized data. Ready for ad-hoc querying.
  • 🥇 Gold (Curated): Business-level aggregates. Highly refined data ready for BI dashboards and Machine Learning.

6. Pipeline Best Practices

# 1. Fail Fast, Fail Loud
# Validate data contracts at the boundaries (e.g., using Great Expectations).
# Do not let corrupt data silently pollute the data lake.

# 2. Separate Compute and Storage
# Rely on architectures where scaling storage (S3) doesn't force you 
# to pay for idle compute, and vice versa.

# 3. Parameterize Everything
# Never hardcode dates. Pass execution dates at runtime so pipelines 
# can be easily backfilled.
def extract(execution_date):
    path = f"s3://raw-data/{execution_date.year}/{execution_date.month}/"
    return spark.read.parquet(path)

# 4. Use Dead Letter Queues (DLQ)
# When a record fails parsing, don't crash the pipeline. Route the bad 
# record to a DLQ for manual inspection, and process the rest.
