Inversion of Control, Beans, ApplicationContext, and AOP Proxies.

1. IoC & Dependency Injection

Inversion of Control (IoC): Instead of objects creating their own dependencies using new, the Spring container (ApplicationContext) injects them. This decouples the code and makes testing easier.

Dependency Injection Types

  • Constructor Injection (Recommended): Ensures dependencies are not null, allows fields to be final, and simplifies unit testing without mocking frameworks.
  • Field Injection (@Autowired): (Anti-pattern) Hard to test outside of Spring, allows circular dependencies which can crash at runtime, and breaks encapsulation.
  • Setter Injection: Good for optional dependencies.
@Service
public class OrderService {
    private final PaymentService paymentService;

    // Constructor Injection (Spring 4.3+ auto-wires if there's only one constructor)
    public OrderService(PaymentService paymentService) {
        this.paymentService = paymentService;
    }
}

2. The Bean Lifecycle

Understanding when a Bean is created and destroyed is critical for advanced configurations.

  1. Instantiation: Spring creates the object instance.
  2. Populate Properties: Spring injects all required dependencies.
  3. BeanPostProcessor (Before): Spring calls postProcessBeforeInitialization. (Often used to wrap beans in proxies).
  4. @PostConstruct: Custom initialization logic runs here.
  5. BeanPostProcessor (After): Spring calls postProcessAfterInitialization.
  6. Ready for Use: The bean is placed in the ApplicationContext.
  7. @PreDestroy: Called when the ApplicationContext is shutting down.

3. Bean Scopes

Singleton (Default) One shared instance per ApplicationContext. Stateless beans (Services, Repositories). Not thread-safe if you store state in fields!
Prototype A new instance is created every time it is injected. Good for stateful objects.
Request One instance per HTTP request (Web applications only).
Session One instance per HTTP session (Web applications only).

4. Aspect-Oriented Programming (AOP)

AOP allows you to separate cross-cutting concerns (like logging, security, or transactions) from the core business logic. Spring achieves this by wrapping your Bean in a dynamically generated Proxy class.

Key Terminology

  • Aspect: The module containing the cross-cutting logic (e.g., a Logging class).
  • Advice: The action taken at a particular point (e.g., "log this before the method runs").
  • Pointcut: A regex-like expression defining where the advice should be applied.
@Aspect
@Component
public class LoggingAspect {

    // Pointcut targeting all methods inside the 'service' package
    @Around("execution(* com.myapp.service.*.*(..))")
    public Object logExecutionTime(ProceedingJoinPoint joinPoint) throws Throwable {
        long start = System.currentTimeMillis();
        
        // Execute the actual business method
        Object proceed = joinPoint.proceed();
        
        long executionTime = System.currentTimeMillis() - start;
        System.out.println(joinPoint.getSignature() + " executed in " + executionTime + "ms");
        
        return proceed;
    }
}

Warning: Because Spring AOP uses proxies, calling a method from within the same class bypasses the proxy! (This is why @Transactional doesn't work if called by a method in the same class).
