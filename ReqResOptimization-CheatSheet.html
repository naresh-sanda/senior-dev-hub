Request & Response Optimization

Author: Naresh Kumar Sanda

Pagination strategies, Payload reduction, and HTTP optimizations.

1. Cursor vs Offset Pagination

When returning large lists, you must paginate. There are two primary strategies.

Offset Pagination (Classic)

Uses LIMIT <size> OFFSET <page * size> (e.g., page=5, size=20).

  • Pros: Easy to implement. Allows jumping to a specific page (e.g., "Page 15").
  • Cons: Terrible performance on deep pages. The database must read all preceding rows just to discard them (e.g., OFFSET 10000 reads 10000 rows). Prone to data duplication if new items are inserted while the user is paging.

Cursor Pagination (Modern/Infinite Scroll)

Uses a unique identifier (cursor), often the ID or timestamp of the last seen item. Query: WHERE id > <last_seen_id> LIMIT <size>.

  • Pros: Extremely fast, even for deep pages, because it utilizes a B-Tree index to jump straight to the correct record. Prevents data duplication during live updates.
  • Cons: Cannot jump to a specific page (must go next/prev). Requires a sequential, indexable column.

2. Payload Compression

Reducing the size of JSON payloads sent over the network saves bandwidth and improves load times.

  • GZIP: The standard algorithm. Supported by almost all clients and servers. High compression ratio with moderate CPU usage.
  • Brotli: Developed by Google. Offers ~20% better compression than GZIP for text/JSON, but requires more CPU to compress. Ideal for static assets compressed ahead of time.

Implementation: Clients send Accept-Encoding: gzip, br header. Servers respond with Content-Encoding: gzip.

3. Client-Side Caching (ETags)

An Entity Tag (ETag) is an identifier (usually a hash) representing a specific version of a resource.

  1. Client requests /api/user/123.
  2. Server responds with data and an ETag header: ETag: "v1-hash-abc".
  3. Later, client requests the same resource, including If-None-Match: "v1-hash-abc".
  4. Server checks if the resource has changed. If not, it returns a 304 Not Modified with an empty body.

Benefit: Saves massive amounts of bandwidth and client rendering time, though the server still has to query the DB to generate the hash.

4. HTTP/2 & HTTP/3

HTTP/2

Replaces HTTP/1.1's text-based protocol with a binary framing layer.

  • Multiplexing: The biggest feature. Multiple requests and responses can be sent concurrently over a single TCP connection, solving the "head-of-line blocking" problem.
  • Server Push: Server can send assets to the client before the client asks for them.
  • Header Compression (HPACK): Compresses HTTP headers.

HTTP/3

Runs over QUIC (a UDP-based protocol) instead of TCP.

  • Eliminates TCP handshakes, dramatically speeding up initial connection times (0-RTT).
  • Solves packet-level head-of-line blocking (if one packet is lost, it doesn't block other parallel streams).
