Spring Data JPA & JDBC

Author: Naresh Kumar Sanda

Entity mapping, JPQL, performance tuning, and solving N+1.

1. Entity Lifecycle (Hibernate)

  • Transient: An object created using new. Not associated with a Hibernate Session. No DB row exists yet.
  • Managed / Persistent: Associated with the Session. Changes to this object will automatically be saved to the database upon transaction commit (Dirty Checking).
  • Detached: The Session was closed. Changes to the object are no longer tracked. You must call save() or merge() to re-attach it.
  • Removed: Scheduled for deletion.

2. Solving the N+1 Query Problem

The Problem: When you fetch a List of 10 Authors, and each Author has a List of Books (Lazy loaded). Iterating through the authors to get their books triggers 1 query for the authors, and 10 queries for the books (1 + N = 11 queries).

Solution 1: EntityGraph

Dynamically tells JPA to fetch the related entities in a single JOIN query, ignoring the default Lazy loading.

public interface AuthorRepository extends JpaRepository<Author, Long> {
    // Eagerly fetches the 'books' collection using a LEFT OUTER JOIN
    @EntityGraph(attributePaths = {"books"})
    List<Author> findAll();
}

Solution 2: JOIN FETCH in JPQL

@Query("SELECT a FROM Author a JOIN FETCH a.books")
List<Author> findAllAuthorsWithBooks();

3. Transaction Management

The @Transactional annotation wraps your method in a database transaction using an AOP Proxy.

@Service
public class TransferService {

    // Rollback ONLY occurs on unchecked exceptions (RuntimeException) by default!
    @Transactional(rollbackFor = Exception.class) 
    public void transferMoney(Long from, Long to, BigDecimal amount) throws CustomCheckedException {
        // ... business logic ...
    }
}

Propagation Types

  • REQUIRED (Default): Join existing transaction. If none exists, create a new one.
  • REQUIRES_NEW: Suspend the current transaction and create a completely new one.
  • SUPPORTS: Run within a transaction if it exists; otherwise, run non-transactionally.

4. Optimistic vs Pessimistic Locking

Optimistic Locking (Best for High Read Volume)

Uses a @Version column. Does not lock the DB row. When saving, Hibernate checks if the version in the DB matches the object's version. If it changed, it throws OptimisticLockException.

@Entity
public class Product {
    @Id private Long id;
    private int quantity;
    @Version private Long version; // Hibernate handles this automatically
}

Pessimistic Locking (Best for High Write Contention)

Issues a SELECT ... FOR UPDATE query, physically locking the row in the database so no other transaction can read/write it until the current transaction finishes.

public interface ProductRepository extends JpaRepository<Product, Long> {
    @Lock(LockModeType.PESSIMISTIC_WRITE)
    @Query("SELECT p FROM Product p WHERE p.id = :id")
    Optional<Product> findByIdWithLock(@Param("id") Long id);
}

5. Pagination & Sorting

Never return millions of rows. Use Spring Data's Pageable.

// Repository
Page<User> findByStatus(String status, Pageable pageable);

// Service
Pageable firstPageWithTwoElements = PageRequest.of(0, 2, Sort.by("lastName").descending());
Page<User> page = userRepository.findByStatus("ACTIVE", firstPageWithTwoElements);

List<User> users = page.getContent();
long totalElements = page.getTotalElements();
int totalPages = page.getTotalPages();
