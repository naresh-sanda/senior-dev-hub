App & DB Performance

Author: Naresh Kumar Sanda

Indexes, EXPLAIN plans, Connection Pools, and Thread Dumps.

1. Database Indexes

Indexes drastically speed up SELECT queries but slow down INSERT, UPDATE, and DELETE operations because the index must be updated alongside the data.

B-Tree Indexes (Default)

Stores data in a balanced tree structure. Excellent for exact matches, ranges (<, >, BETWEEN), and sorting (ORDER BY).

Hash Indexes

Only useful for exact matches (=). Cannot be used for range queries or sorting.

Composite Indexes

An index on multiple columns (e.g., INDEX(last_name, first_name)). Rule of thumb: The order matters! It works like a phone book. It can speed up queries filtering by last_name, or last_name AND first_name, but it is USELESS for queries filtering ONLY by first_name.

2. Query EXPLAIN Plans

Prefixing a query with EXPLAIN (or EXPLAIN ANALYZE in PostgreSQL/MySQL) shows how the DB engine executes the query.

Index Scan / Index Seek ✅ The DB found a relevant index and used it to find the rows quickly. This is what you want.
Full Table Scan / Seq Scan ❌ The DB had to read every single row in the table. Terrible for large tables. Indicates a missing index.
Filesort ❌ (MySQL) The DB had to write data to a temporary file on disk to sort it because the index wasn't sufficient. Very slow.

3. Connection Pooling (HikariCP)

Opening a database connection involves network handshakes and authentication, which is extremely slow. A Connection Pool keeps a set of connections open and reuses them.

HikariCP Tuning (Spring Boot Default)

  • maximumPoolSize: How many connections can be open at once? Surprisingly, smaller is often better. A formula from PostgreSQL experts: connections = ((core_count * 2) + effective_spindle_count). Usually, 10-20 is plenty for a single microservice node.
  • minimumIdle: HikariCP recommends leaving this same as maximumPoolSize for fixed-size pools to avoid sudden connection drops/creates.
  • connectionTimeout: How long a thread waits for a connection before throwing an exception (default 30s).

4. Application-Level Tuning

  • N+1 Query Problem: Mentioned in the JPA cheat sheet, but it's the #1 cause of ORM slowness. Always use JOIN FETCH or EntityGraphs to eagerly fetch required relations.
  • Thread Dumps: If your Java application freezes (but CPU is low), take a thread dump (jstack <pid>). Look for threads in the BLOCKED state. This often indicates a thread waiting on a database connection pool that is exhausted, or a synchronized block bottleneck.
  • Async Processing: Do not block HTTP threads for slow operations (like sending an email). Delegate to an ExecutorService, Virtual Threads, or a Message Queue (Kafka/RabbitMQ).
