App & DB Performance

Author: Naresh Kumar Sanda

Query EXPLAIN plans, Indexes, and HikariCP Tuning.

1. Using EXPLAIN

Prefix your slow query with EXPLAIN to see exactly how MySQL executes it. Look closely at the type column. If it says ALL, MySQL is doing a Full Table Scan (terrible). If it says ref or eq_ref, it's using an index (good!).
EXPLAIN SELECT * FROM users WHERE email = 'test@test.com';

2. B-Tree Indexes

Indexes are separate data structures (B-Trees) that keep a sorted copy of a column, pointing back to the main table rows. They make SELECT extremely fast, but slow down INSERT/UPDATE (because the tree must be re-sorted).

-- Composite Index (Order matters!)
CREATE INDEX idx_last_first ON users(last_name, first_name);
-- This index works for "WHERE last_name='Smith'"
-- It DOES NOT work for "WHERE first_name='John'"

3. HikariCP (Connection Pooling)

Opening a new DB connection for every request takes hundreds of milliseconds (TCP handshake, SSL). A Connection Pool keeps a pool of connections permanently open and reuses them!

Golden Rule: The optimal pool size is usually much smaller than you think: Connections = ((core_count * 2) + effective_spindle_count). Usually around 10-20 connections per app instance is perfect.
