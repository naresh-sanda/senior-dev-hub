API Security & Authorization

Author: Naresh Kumar Sanda

Comprehensive guide to Basic, Bearer, Session, JWT, and OAuth2 architectures.

1. Authentication vs. Authorization

  • Authentication (AuthN): Who are you? (e.g., verifying a username and password).
  • Authorization (AuthZ): What are you allowed to do? (e.g., checking if the user has the ROLE_ADMIN permission).

2. No Auth (Public/Anonymous)

Some endpoints explicitly require NO authentication. The server processes the request without checking any headers.

  • Use Cases: /actuator/health (health checks for Load Balancers/K8s), Public product catalogs, or user registration endpoints.
  • Spring Security Example: http.authorizeRequests().antMatchers("/public/**").permitAll();

3. Basic Authentication

Warning: Basic Auth sends credentials encoded (NOT encrypted) in the header. It is fundamentally insecure unless sent over HTTPS (TLS/SSL).

The client concatenates the username and password with a colon (user:pass), Base64 encodes it, and sends it in the Authorization header on EVERY request.

# Client Request Header
Authorization: Basic QWxhZGRpbjpvcGVuIHNlc2FtZQ==

Pros: Extremely simple to implement. Cons: Insecure over HTTP, requires sending the actual password on every single request.

4. Session-Based Authentication (Stateful)

The traditional approach for web browsers. The server does the heavy lifting.

  1. User logs in with credentials.
  2. Server validates, creates a "Session Object" in server memory (or Redis), and sends a Session ID cookie (like JSESSIONID) back to the browser.
  3. Browser automatically sends the Cookie on subsequent requests.
  4. Server looks up the Session ID in its memory to identify the user.
Drawbacks: 1. Scalability: If you have 3 backend servers, the session must be shared (e.g., via Redis) otherwise a user hitting Server 2 will be "logged out". 2. Vulnerability: Susceptible to CSRF (Cross-Site Request Forgery) attacks.

5. Bearer Tokens & JWT (Stateless)

Unlike Sessions, JWT (JSON Web Token) is Stateless. The server does NOT save the token in a database or memory.

A JWT consists of 3 Base64-encoded parts separated by dots: Header.Payload.Signature.

  • Header: Specifies the algorithm used (e.g., HS256 or RS256).
  • Payload: The actual data (Claims) like {"userId": 123, "role": "ADMIN"}. Anyone can decode and read this, so do not put passwords here!
  • Signature: The crucial security piece. The server hashes the Header + Payload using a secret key.

How it works:

When the server receives a request with Authorization: Bearer <token>, it recalculates the Signature using its secret key. If the signatures match, the token is perfectly valid and hasn't been tampered with. No database lookup is required!

# Client Request Header
Authorization: Bearer eyJhbGciOiJIUzI1Ni...[rest of JWT]

6. OAuth 2.0 & OpenID Connect (Delegated Access)

OAuth 2.0 is an Authorization framework. OpenID Connect (OIDC) is an identity layer built on top of OAuth 2.0 to provide Authentication.

The standard "Login with Google/GitHub" flow (Authorization Code Flow):

  1. User clicks "Login". Your App redirects them to the Authorization Server (Google).
  2. User logs into Google and grants permission (Consent).
  3. Google redirects the user back to your App with a short-lived Authorization Code in the URL.
  4. Your Backend securely contacts Google (server-to-server) and trades the Code + Client Secret for an Access Token (usually a JWT) and a Refresh Token.
  5. Your App uses the Access Token to fetch resources on behalf of the user.

7. API Keys (Machine-to-Machine)

Typically used for server-to-server communication or monetizing public APIs (e.g., Google Maps API). The client sends a unique, long-lived string (the key) in a custom header (e.g., X-API-Key).

Use Cases: Rate-limiting, tracking usage/billing per client. Not suitable for authenticating individual users in a web browser because the key is permanent and hard to secure on the client side.
