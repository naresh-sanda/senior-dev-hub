1. Cluster Architecture

  • Driver Node: Runs the `main()` method of the application. It creates the SparkSession, translates user code into actual jobs, builds the Directed Acyclic Graph (DAG), and schedules tasks across workers.
  • Cluster Manager: Allocates physical resources (CPU/Memory) across applications. Examples: YARN, Mesos, Kubernetes, or Spark Standalone.
  • Worker Node / Executors: Processes that run on worker nodes. They execute the tasks sent by the Driver and hold data in memory or disk across them.

2. RDD vs DataFrame vs Dataset

RDD (Resilient Distributed Dataset) The low-level, fundamental data structure. Distributed collection of Java/Python objects. Hard to optimize because Spark doesn't understand the structure of the data inside the object. Usually avoided in modern Spark code.
DataFrame An RDD organized into named columns (like a relational DB table). High performance because Spark knows the exact schema and can optimize execution plans. Untyped at compile-time (errors happen at runtime).
Dataset (Java/Scala only) Provides the benefits of RDDs (compile-time type safety) and DataFrames (schema-based performance optimization). Not available in Python because Python is dynamically typed.

3. Catalyst Optimizer & Tungsten

These two engines are why DataFrames are significantly faster than raw RDDs.

  • Catalyst Optimizer: An extensible query optimizer. It takes your code, converts it into an Unresolved Logical Plan, applies schema checks to make a Logical Plan, optimizes it (e.g., pushing filters down to the database before joining), and generates multiple Physical Plans to choose the cheapest one.
  • Project Tungsten: Focuses on hardware optimization (CPU & Memory). It manages memory explicitly (avoiding Java Garbage Collection overhead), uses binary processing, and generates optimized bytecode at runtime (Whole-Stage Code Generation).

4. Managing Shuffles

A "Shuffle" occurs when data needs to be redistributed across the cluster's network. It involves writing data to disk, sending it over the network, and reading it back. It is the most expensive operation in Spark.

Operations that trigger Shuffles:

  • groupByKey(), reduceByKey(), agg()
  • join() (unless one side is a Broadcast Join)
  • repartition() (but NOT coalesce())
  • distinct()

Optimization: Use spark.sql.shuffle.partitions to tune the number of partitions created during a shuffle. The default is 200, which is often too high for small datasets and too low for massive datasets. Aim for ~128MB per partition.
