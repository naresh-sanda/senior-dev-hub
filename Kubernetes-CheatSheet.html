Kubernetes (K8s)

Author: Naresh Kumar Sanda

Master orchestration: Pods, Deployments, Services, and Configuration.

1. K8s Architecture

Control Plane (Master) Manages the cluster. Components: kube-apiserver (front-end REST API), etcd (key-value store for cluster state), kube-scheduler (assigns pods to nodes), kube-controller-manager (handles node failures, replication).
Worker Node Runs the applications. Components: kubelet (agent ensuring containers run), kube-proxy (network routing), and a Container Runtime (containerd, docker).
Pod The smallest deployable unit. Usually contains one container, but can hold multiple tightly coupled containers (e.g., sidecars) sharing the same network/storage.

2. Workloads

Deployment

Manages stateless applications. Ensures a specified number of Pod replicas are running. Enables rolling updates and rollbacks.

apiVersion: apps/v1
kind: Deployment
metadata:
  name: nginx-deployment
spec:
  replicas: 3
  selector:
    matchLabels:
      app: nginx
  template:
    metadata:
      labels:
        app: nginx
    spec:
      containers:
      - name: nginx
        image: nginx:1.14.2
        ports:
        - containerPort: 80

StatefulSet vs DaemonSet

  • StatefulSet: Used for stateful apps (databases). Provides stable, unique network identifiers and persistent storage for each Pod. Pods are created sequentially.
  • DaemonSet: Ensures that ALL (or some) Nodes run a copy of a specific Pod. Usually used for logs collection (Fluentd) or node monitoring.

3. Services & Networking

Pods are ephemeral; their IP addresses change when they are recreated. A Service provides a stable IP address and DNS name to access Pods.

ClusterIP (Default) Exposes the Service on a cluster-internal IP. Only reachable from within the cluster.
NodePort Exposes the Service on each Node's IP at a static port (between 30000-32767). Reachable from outside the cluster.
LoadBalancer Provisions a cloud provider's external load balancer (AWS ALB, GCP TCP Load Balancer) pointing to the NodePorts.
Ingress Not a service type, but an API object that manages external HTTP/HTTPS access to Services. Provides path-based routing, SSL termination, and virtual hosting.

4. ConfigMaps & Secrets

Configuration should be decoupled from image contents to make applications portable.

apiVersion: v1
kind: ConfigMap
metadata:
  name: app-config
data:
  DB_HOST: "mysql-service.default.svc.cluster.local"
  APP_ENV: "production"
---
apiVersion: v1
kind: Secret
metadata:
  name: db-credentials
type: Opaque
data:
  # Values MUST be base64 encoded!
  password: cGFzc3dvcmQxMjM=

Injecting them into a Pod:

containers:
  - name: myapp
    image: myapp:1.0
    env:
      - name: DB_HOST
        valueFrom:
          configMapKeyRef:
            name: app-config
            key: DB_HOST
      - name: DB_PASSWORD
        valueFrom:
          secretKeyRef:
            name: db-credentials
            key: password

5. Essential kubectl Commands

# Context and Namespaces
kubectl config current-context         # Show current cluster
kubectl get namespaces                 # List all namespaces
kubectl config set-context --current --namespace=my-namespace # Change default namespace

# Managing Resources
kubectl apply -f manifest.yaml         # Create or update resources
kubectl delete -f manifest.yaml        # Delete resources
kubectl get pods -w                    # List pods and watch (-w) for changes
kubectl get svc,deploy,pods            # List multiple resource types
kubectl describe pod <pod-name>        # Detailed information (great for debugging CrashLoopBackOff)

# Debugging
kubectl logs <pod-name> -f             # Tail logs
kubectl logs <pod-name> -c <container> # Tail logs for specific container in a multi-container pod
kubectl exec -it <pod-name> -- /bin/sh # Open an interactive shell in a pod
kubectl port-forward svc/my-svc 8080:80 # Forward local port 8080 to service port 80
